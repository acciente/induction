What is dragonFly MVC?

dragonFly MVC is a Java(tm) web application development framework. The goal of dragonFly is to simplify the development of high-performance, maintainable web applications using Java(tm). dragonFly MVC is a 100% pure Java(tm) implementation that runs inside a servlet container.

Other popular web applications frameworks for Java that are may be compared to dragonFly include Spring MVC, Struts, Tapestry, WebWork and JPublish.

What motivated dragonFly MVC?

There is already a large number of web application frameworks for Java, so why another Java web application framework? Very simply, it was felt that there is still room for significant improvement.

Following are some of the specific motivations for dragonFly:

  • It should not be necessary to redeploy an application every time Java code is changed. It should be possible to simply recompile the changed source (hopefully using your favorite IDE) and simply reload the respective web page.

  • It should be possible to use mature Java dependency analysis tools in an IDE to analyze the dependencies between models, views and controllers (For example: Which controllers use this model object? What controllers will be impacted if I edit this template?)

  • An application should not be cluttered by superfluous XML configuration files.

The measure of improvement is perhaps ultimately subjective, and there are many perspectives to a problem, so feedback from the community at large is greatly appreciated.

dragonFly MVC quick user guide

dragonFly MVC advocates the use of the Model-View-Controller (MVC) paradigm to web application development. Following is a brief of the summary of how dragonFly MVC supports models, views and controllers.

In dragonFly MVC you will write the important types of classes you will write are controllers, views and models. A very simple application may only use controllers, but most applications will use controllers, views and models.

  • Controllers
    • All requests are processed by a controller

    • A controller is a POJO which implements the Controller marker interface (the Controller interface just marks a class as a controller, and does not enforce any methods)

    • A model object or environment object (such as the Request, Response or Form) is accessible in a controller by simply declaring a parameter of the model object's type in the controler's respective method (this is referred to "parameter injection" in dragonFly)

    • A controller may have a method for each distint task that it performs
      • The default URL to controller mapper, allows specifying the controller class and method name in the path to invoke a controller, e.g: http://www.dragonflymvc.org/myapp/controller/Signup/save, where the controller class is myapp.controller.Signup and a method named "save" should be invoked on the class
      • Omitting the method cause an assumed (configurable) method name to be called

    • The type of the return value of a controller is significant, each possible type and their meaning is given below:
      • Template: processes and return the content of the specified template object to the client
      • Text: returns the text contents to the client
      • Image/ImageStream: returns the image contents to the client
  • Views
  • Models
- views - a view class is a POJO that implements one of the following interfaces (based on the type of view): - Template - Image - Text - ImageStreamer - a class, say C1, that implements the Template interface uses one of the integrated template engines - when processing the template the only data passed to the template is an instance of the class C1 - so C1 serves as the bean that defines clearly what data is uses by the associated template - models - a model is any user defined class - a configurable factory may be provided for each model class - declarative control of the lifecyle of each model class - application: a model object is created for the life of the application - session: a model object is created for each browser session - local: a model object is created for each controller invocation

dragonFly MVC technical overview

  • General
    • Runtime class reloading for controllers, views and models (no more redeploy/restart)

    • Ability to analyze dependencies between controllers, views and models using basic java dependency analysis tools in any IDE

    • Extensibile architecture
      • Pluggable configuration loader
      • Pluggable URL to controller mapper
      • Pluggable templating engines (initial support for Freemarker, adding other templating engines should be very easy, Velocity is likely to be next)

  • Controller
    • Parameter injection: a model object is accessible in a controller by simply declaring a parameter of the model object's type in the controler's respective method

    • Sophisticated form parameter processing, to simpify using list and map structures in an HTML form

    • Handling an uploaded file should be as simple as handling a string input on a form (this would not have been mentioned expect for how hard some frameworks make handling uploaded files)

  • View
    • Integrated support for templating engines (initial support for Freemarker)

    • Each template view is represented by a class that encapuslates all the data the template depends on, enabling powerful Java based dependency analysis when template changes need to be made to text-based HTML templates

  • Model
    • Declarative control of the lifecyle of model objects
