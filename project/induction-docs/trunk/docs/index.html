What is Induction?

Induction is a Java(tm) web application development framework. The goal of Induction is to simplify the task of building complex, high-performance, maintainable web applications using Java(tm). Induction is a 100% pure Java(tm) implementation that runs inside a Servlet container.

Other popular web applications frameworks for Java that may be compared to Induction include Spring MVC, Struts, Tapestry, WebWork and JPublish.

What Motivated Induction?

There is already a large number of web application frameworks for Java, so why another Java web application framework? Very simply, it was felt that there was still room for significant improvement. Following are some of the specific motivations for Induction:

·         It should not be necessary to redeploy an application to a Servlet container every time Java code is changed. It should be possible to simply recompile the changed source (using your favorite IDE) and simply reload the respective web page.

·         It should be possible to use mature Java dependency analysis tools in an IDE to analyze the dependencies between models, views and controllers (For example to answer question such as: Which controllers use this model object? What controllers will be impacted if I edit this template?)

·         An application should not be cluttered by superfluous XML configuration files.

The measure of improvement is perhaps ultimately subjective, and there are many perspectives to a problem, therefore feedback from the community at large is greatly appreciated.

Induction Features Overview

·         General

o        Runtime class reloading for controllers, views and models (no more redeploy/restart)

o        Ability to analyze dependencies between controllers, views and models using basic java dependency analysis tools in any IDE

o        Extensible architecture

§         Pluggable configuration loader

§         Pluggable URL to controller resolver

§         Pluggable templating engines (initial support for Freemarker, adding other templating engines should be very easy, Velocity is likely to be next)

·         Controller

o        Parameter injection: a model object is accessible in a controller by simply declaring a parameter of the model object's type in the controller's respective method

o        Sophisticated form parameter processing, to simplify using list and map structures in an HTML form

o        Handling an uploaded file should be as simple as handling a string input on a form (this would not have been mentioned expect for how hard some frameworks make handling uploaded files)

·         View

o        Integrated support for templating engines (initial support for Freemarker)

o        Each template view is represented by a class that encapsulates all the data the template depends on, enabling powerful Java based dependency analysis when template changes need to be made to text-based HTML templates

·         Model

o        Declarative control of the lifecycle of model objects

Induction Model-View-Controller Overview

Induction advocates the use of the Model-View-Controller (MVC) paradigm to web application development. Following is a brief of the summary of how Induction supports models, views and controllers. In Induction the important types of classes you will write are controllers, views and models and model factories. A very simple application may only use controllers, but most applications will use controllers, views and models and likely on one or more model factories.

·         Controllers

o        All requests received by the Servlet container are processed by a controller

o        A controller is a class which implements the Controller marker interface (the Controller interface just marks a class as a controller, and does not enforce any methods)

o        A model object or environment object (such as the Request, Response or Form) is accessible in a controller by simply declaring a parameter of the model object's type in the controller's respective method (this is basically dependency injection)

o        A controller may have a method for each distinct task that it performs:

§         The default URL to controller resolver, allows specifying the controller class and method name in the path to invoke a controller,
e.g.: http://www.inductionframework.org/demoapp/counter_app/CounterController/incrementCounter, where the controller class is demoapp.counter_app.CounterController and a method named " incrementCounter" should be invoked on the class

§         Omitting the method implies an assumed (configurable) method name should be called (usually named handler)

o        The type of the return value of a controller is significant, each possible type and their meaning is given below:

§         Template: processes and return the content of the specified template object to the client

§         Text: returns the text contents to the client

§         Image/ImageStream: returns the image contents to the client

·         Views

o        A view class is a that implements one of the following interfaces (based on the type of view):

§         Template

§         Image

§         ImageStreamer

§         Text

o        A class, say C1, that implements the Template interface uses one of the integrated template engines

§         When processing the template the only data passed to the template is an instance of the class C1, C1 serves as the bean that defines clearly what data is uses by the associated template

·         Models

o        Implements pure application logic, completely independent of views and controllers

o        A model is any user defined class

o        A configurable factory may be provided for each model class

o        Declarative control of the lifecycle of each model class:

§         Application: a model object is created for the life of the application

§         Session: a model object is created for each browser session

§         Local: a model object is created for each controller invocation
